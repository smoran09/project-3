OOP terms and definitions

  • Factory- allows an interface or a class to create an object, but lets subclasses decide which class or object to instantiate.
  • Instantiation/instantiating- is creating a copy of the class which inherits all class variables and methods.
  • Object- An instance of a class that is created at runtime. In C#, an instance is created with the new keyword.
  • Class-The code describing a particular entity in the application, such as Customer, Order, or Address.
  • Namespace- provide a method for preventing name conflicts in large projects.
  • Constructor- Code that is executed each time an instance of the class is created.
  • Fixture- a defined, reliable and consistent context for the tests.

Continued

  • Type hint- a formal solution to statically indicate the type of a value within your Python code.
  • Type cast- changing an variable of one data type into another.
  • Unit test- a type of software testing where individual units or components of a software are tested.
  • Static class- Class that cannot be instantiated, so no objects can be created from the class. The class members are instead accessed using the class name.
  • Static-method- Adding the static modifier on a member of the class (property or method) defines that the member belongs to the class itself, not an instance of the class.
  • Class method- The code describing a particular entity in the application, such as Customer, Order, or Address.
  • Facade- hides the complexities of the system and provides an interface to the client using which the client can access the system.
  • Design patterns used- Via link Tutorials included
  • Method- The code defining the actions or behaviors of the class, such as Validate or CalculateTotal.
  • Property- The code defining the data managed by the class, such as CustomerName, OrderNumber, or EmailAddress.

Additional Terms

  • Inheritance- A relationship between classes whereby child (or derived) classes inherit all of the members of the parent (or base) class.
  • Encapsulation- A way to hide (or encapsulate) the data and implementation details within a class, thus hiding complexity.
